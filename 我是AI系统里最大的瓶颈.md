# 我是AI系统里最大的瓶颈

Author: @v5ria WK

最近和朋友聊天，说了一句话：**我现在绝对自己就是阻碍。**

说出来有点讽刺，但越想越觉得是真的。

## 问题从哪里来

我们在做一个需要大量AI交互的系统。理论上，AI对话要发生几万次，每一轮都是迭代推进的机会。但现实是，这几万次对话里，大多数的效率太低了。

不是AI的问题。AI越来越聪明，响应速度快，几十秒就能给出结果。问题出在人这一侧。

人参与进来以后，大多数时间在**学习**，不是在推进。在理解AI的输出，在消化，在思考下一步怎么说，在犹豫要不要接受这个结果。这个过程本身消耗了大量时间，而且水平不高的参与者，还会把方向带偏，引入不必要的确认循环。

结果就是：人的参与，慢慢成了系统迭代速度的瓶颈。

## 风向变了

我注意到一个趋势：越来越多的人开始讨论**去掉人的参与**。

不是说人不重要，而是说在系统迭代这件事上，理想的状态是：AI自己设计目标，自己执行，自己检查，自己反省，自己再定规划来优化。形成一个完整的自我迭代闭环。

人只需要在起点给出方向，在终点验收结果。中间那些低效的确认、审批、修正循环，能省就省。

回顾一下 AI 辅助开发的演变，大致经历了三个阶段：

1. **补全时代**：AI 做自动补全，人一个字符一个字符地敲，AI 帮你猜下一步。
2. **同步 Agent 时代**：人通过"提示—响应"的循环指挥 Agent，Agent 能处理更复杂的任务，但每一步都需要人实时参与。
3. **自主 Agent 时代**：Agent 独立运行，在更长的时间尺度上完成更大的任务，需要更少的人为干预。人的角色从"逐行指导"变成"定义问题 + 审阅成果"。

第三个阶段正在到来，而且来得比大多数人预期的快。

**同步交互本身就是瓶颈。** 你坐在那里盯着 Agent 的输出，等它生成，看一遍，给意见，等它改——这个过程里你的大脑在做的事其实大部分是"确认"，不是"创造"。而且人坐在那里，同一时间能跑的 Agent 数量也就那么几个。

更好的模式是让 Agent 在隔离的环境里独立运行，开发者把任务交出去，然后转去做别的事。Agent 持续工作数小时，反复迭代和测试，完成后以便于快速审阅的形式返回结果——日志、截图、实时预览，而不是让你去看每一行 diff。

这个判断让我有点焦虑，因为它意味着，**在AI系统里，人能创造价值的空间正在收窄**，至少在"执行"这个层面上是这样的。

## 我上周做了个实验

上周末做了个AI考试系统的项目。我给AI设计了一个任务，让它自我迭代。

它跑了两天。

两天以后拿出来，各方都很满意。

我几乎没有介入中间过程。这两天它自己在做什么，我基本上是事后才看的。

这个结果让我一方面很兴奋，另一方面更焦虑了——如果AI两天可以完成这件事，而我在的话可能反而会打断它、拖慢它，那我的价值在哪里？

这次实验让我意识到，走在前面的那批开发者已经在这样工作了：

- 几乎 100% 的代码都由 Agent 编写
- 人的时间花在**拆解问题、审查结果、提供反馈**上
- 同时启动多个 Agent 并行工作，而不是手把手引导单个 Agent 直至完成

你不再是在写代码，而是在搭建一座**生产软件的工厂**。这座工厂由一支 Agent 组成，你的工作是给出方向、配备工具、审阅成果。

## 范式是什么

我现在每天想的是，怎么给AI设计出更好的任务，让它能够自我迭代更多次，工作更久，产出更多有价值的东西。

回头看考试系统这个实验，我觉得它能跑起来的核心原因是：**"考试"本身有客观答案，AI可以自己跑、自己评分、自己知道迭代结果是更好还是更差。评价标准是内嵌在任务里的。**

这就是可以自迭代的任务的本质特征。

所以判断一个任务是否适合AI自迭代，其实只需要问一个问题：

> **成功的标准，能不能被机器自动验证？**

- 能 → 可以自迭代，人只需要在起点定义目标，终点验收
- 不能（需要人的主观判断）→ 人仍然是必要的，但要把介入压缩到最小

基于这个判断，我目前摸索出几个具体做法：

**1. 把验收标准前置、量化**

不要说"做一个好的推荐系统"，要说"做一个推荐系统，点击率在测试集上超过baseline 5%"。目标越具体，AI自我判断的空间越大，找人确认的次数越少。

**2. 给AI一个可以反复跑的测试环境**

代码有单测，内容有评分rubric，数据有schema校验。这是自迭代的基础设施。没有这个，AI每次改完都要问你"这样对吗"，你又变成瓶颈了。

理想的状态是给 Agent 一个完全隔离的沙箱：可以自由安装依赖、跑测试、构建项目，不影响你的本地环境。Agent 在这里反复迭代，直到所有测试通过、输出足够可靠，再把结果交回给你。

**3. 把自己的介入点从"过程review"变成"目标设定 + 方向纠偏"**

以前的模式：AI给出中间结果 → 我看 → 我给意见 → AI继续。  
更好的模式：在最开始花更多时间想清楚目标和边界 → AI自跑 → 只在它卡住或跑偏时介入。

前期多花一倍时间定义清楚，换来中间完全不介入。

**4. 用工件代替实时盯盘**

不要盯着 Agent 的实时输出，而是让它完成后给你一份可以快速审阅的工件——日志摘要、关键截图、结果对比。这样你才能同时跑多个 Agent，因为你不可能同时盯三个 Agent 的实时过程，但你可以在它们都完成后，花几分钟浏览三份工件，挑出最好的。

**5. 并行而非串行**

一次只跑一个 Agent，坐在那里等它完成，本质上还是人在卡节奏。更好的方式是同时启动多个 Agent，让它们并行处理不同的任务，或者对同一个任务跑多个方案，最后从结果里挑最好的。

这改变了"效率"的定义：不是单次迭代更快，而是**单位时间内并行的迭代更多**。

## 人真正的价值在哪里

想清楚范式之后，我对"人的价值"这件事也有了新的理解。

人不应该是过程中的reviewer，而应该是：

- **定义问题的人**：什么是值得解决的问题，优先级怎么排
- **设计评价标准的人**：成功长什么样，用什么衡量
- **识别跑偏的人**：AI的迭代方向对不对，有没有在优化错误的指标

这三件事目前AI还替代不了，因为它们需要对业务和用户的深度理解。

把精力集中在这里，人就不是瓶颈，而是杠杆。你的价值不在于亲手砌每一块砖，而在于设计这座工厂的蓝图，并确保流水线不会跑偏。

## 今天想明白的另一件事

人在流程中间还有一个更深层的问题，不只是速度慢。

**人的参与会强迫 AI 的流程"降维"到人能理解的层面。**

AI 在自主迭代的时候，它设计的中间步骤不需要对任何人解释，不需要符合人类的工作习惯，不需要遵循"先做 A 再做 B"这种人能看懂的线性逻辑。它可以同时探索几十条路径，在人看来毫无章法的顺序里完成任务，走出人类永远不会主动选择的捷径——不是因为那条路错了，而是因为那条路人类看不懂，所以人类不敢走。

一旦人介入，这一切就会被打断。人需要理解当前在哪一步，需要确认下一步是否合理，需要 AI 把思路解释清楚。这个"解释"的动作本身，就会把 AI 的流程锁定在人类可理解的范围里。效率并不是因为解释慢了，而是因为**可理解性本身就是一种约束**，它剪掉了那些虽然有效但无法被解释的可能性。

这让我重新理解了为什么"结果验收"比"过程参与"更重要。

你不需要看懂 AI 是怎么走到这个结果的，就像你不需要看懂压缩算法的内部逻辑，只需要知道文件能还原、数据没有丢失。你需要的是一套好的验收标准，能判断结果对不对——而不是一套好的审查流程，确保每一步都在你的视野里。

换句话说：**对过程的控制欲，是另一种形式的瓶颈。** 放弃理解过程，专注于定义目标和验收结果，才是真正把人从循环中解放出来的方式。

## 还有多少时间

这件事的紧迫性可能比大多数人想的要高。

从我观察到的数据来看，Agent 的使用量正在以极快的速度增长，而且已经开始反超传统的逐行编写模式。留给我们适应新范式的窗口期很短——现在不开始思考"怎么让 Agent 自主工作"，等这个模式成为行业标准的时候，你可能连焦虑的资格都没有了，因为你已经不在这个生产循环里了。

## 结尾

如果非要总结一句话的感悟：

**以前我们焦虑的是"AI会不会抢走我的工作"，现在我焦虑的是"我会不会拖累AI的工作"。**

这个转变，比我预想的来得更快。这不是某个人的感觉，而是整个行业正在发生的结构性变化。唯一的问题是：你是在搭建工厂，还是还在手工砌砖？
