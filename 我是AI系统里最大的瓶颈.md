# 我是AI系统里最大的瓶颈

Author: @v5ria WK

最近和朋友聊天，说了一句话：**我现在绝对自己就是阻碍。**

说出来有点讽刺，但越想越觉得是真的。

## 问题从哪里来

我们在做一个需要大量AI交互的系统。理论上，AI对话要发生几万次，每一轮都是迭代推进的机会。但现实是，这几万次对话里，大多数的效率太低了。

不是AI的问题。AI越来越聪明，响应速度快，几十秒就能给出结果。问题出在人这一侧。

人参与进来以后，大多数时间在**学习**，不是在推进。在理解AI的输出，在消化，在思考下一步怎么说，在犹豫要不要接受这个结果。这个过程本身消耗了大量时间，而且水平不高的参与者，还会把方向带偏，引入不必要的确认循环。

结果就是：人的参与，慢慢成了系统迭代速度的瓶颈。

## 风向变了

我注意到一个趋势：越来越多的人开始讨论**去掉人的参与**。

不是说人不重要，而是说在系统迭代这件事上，理想的状态是：AI自己设计目标，自己执行，自己检查，自己反省，自己再定规划来优化。形成一个完整的自我迭代闭环。

人只需要在起点给出方向，在终点验收结果。中间那些低效的确认、审批、修正循环，能省就省。

Cursor 团队最近把这个趋势总结成了**软件开发的三个时代**：

1. **Tab 时代**：AI 做自动补全，人一个字符一个字符地敲，AI 帮你猜下一步。这个时代持续了将近两年。
2. **同步 Agent 时代**：人通过"提示—响应"的循环指挥 Agent，Agent 能处理更复杂的任务，但每一步都需要人实时参与。这个时代可能撑不到一年。
3. **自主 Agent 时代**：Agent 在云端独立运行，在更长的时间尺度上完成更大的任务，需要更少的人为干预。人的角色从"逐行指导"变成"定义问题 + 审阅成果"。

第三个时代正在到来。Cursor 内部合入的 PR 中，已有超过 35% 是由在云端虚拟机上自主运行的 Agent 创建的。他们预计，一年后绝大多数开发工作都会由这类 Agent 完成。

这和我的感受完全一致。**同步交互本身就是瓶颈。** 你坐在那里盯着 Agent 的输出，等它生成，看一遍，给意见，等它改——这个过程里你的大脑在做的事其实大部分是"确认"，不是"创造"。而且人坐在那里，Agent 和人在同一台机器上争抢资源，同一时间能跑的 Agent 数量也就那么几个。

云端 Agent 消除了这两个限制。每个 Agent 跑在自己的虚拟机上，开发者可以把任务交出去，然后转去做别的事。Agent 会持续工作数小时，反复迭代和测试，直到足够有信心，再以便于快速审阅的形式返回结果——日志、视频录像、实时预览，而不是让你去看 diff。

这个判断让我有点焦虑，因为它意味着，**在AI系统里，人能创造价值的空间正在收窄**，至少在"执行"这个层面上是这样的。

## 我上周做了个实验

上周末做了个AI考试系统的项目。我给AI设计了一个任务，让它自我迭代。

它跑了两天。

两天以后拿出来，各方都很满意。

我几乎没有介入中间过程。这两天它自己在做什么，我基本上是事后才看的。

这个结果让我一方面很兴奋，另一方面更焦虑了——如果AI两天可以完成这件事，而我在的话可能反而会打断它、拖慢它，那我的价值在哪里？

Cursor 团队观察到的那些"新型开发者"的工作方式，和我这次的体验几乎一样：

- 几乎 100% 的代码都由 Agent 编写
- 人的时间花在**拆解问题、审查结果、提供反馈**上
- 同时启动多个 Agent 并行工作，而不是手把手引导单个 Agent 直至完成

他们把这个叫做**"工厂"隐喻**：开发者不再主要用来写代码，而是搭建一座"生产软件的工厂"。这座工厂由一支 Agent 车队组成，开发者像和队友协作一样，给出初始方向，为 Agent 配备能独立完成工作的工具，然后审阅成果。

## 范式是什么

我现在每天想的是，怎么给AI设计出更好的任务，让它能够自我迭代更多次，工作更久，产出更多有价值的东西。

回头看考试系统这个实验，我觉得它能跑起来的核心原因是：**"考试"本身有客观答案，AI可以自己跑、自己评分、自己知道迭代结果是更好还是更差。评价标准是内嵌在任务里的。**

这就是可以自迭代的任务的本质特征。

所以判断一个任务是否适合AI自迭代，其实只需要问一个问题：

> **成功的标准，能不能被机器自动验证？**

- 能 → 可以自迭代，人只需要在起点定义目标，终点验收
- 不能（需要人的主观判断）→ 人仍然是必要的，但要把介入压缩到最小

基于这个判断，我目前摸索出几个具体做法：

**1. 把验收标准前置、量化**

不要说"做一个好的推荐系统"，要说"做一个推荐系统，点击率在测试集上超过baseline 5%"。目标越具体，AI自我判断的空间越大，找人确认的次数越少。

**2. 给AI一个可以反复跑的测试环境**

代码有单测，内容有评分rubric，数据有schema校验。这是自迭代的基础设施。没有这个，AI每次改完都要问你"这样对吗"，你又变成瓶颈了。

Cursor 在这一点上做得很彻底：每个云端 Agent 跑在独立的虚拟机上，有自己的沙箱环境，可以自由地安装依赖、跑测试、构建项目，不影响你的本地环境。Agent 可以在这个隔离环境里反复迭代数小时，直到所有测试通过、输出足够可靠。这就是"可以反复跑的测试环境"的工业化版本。

**3. 把自己的介入点从"过程review"变成"目标设定 + 方向纠偏"**

以前的模式：AI给出中间结果 → 我看 → 我给意见 → AI继续。  
更好的模式：在最开始花更多时间想清楚目标和边界 → AI自跑 → 只在它卡住或跑偏时介入。

前期多花一倍时间定义清楚，换来中间完全不介入。

**4. 让AI自己写"反省日志"，用工件代替实时盯盘**

在任务设计里加一步：每次迭代结束，AI要输出"这次做了什么、效果如何、下次打算怎么改"。这个结构化反省是自迭代质量的保障。

Cursor 的做法更进一步：Agent 结束后不只是给你一个 diff，而是生成**日志、视频录像、实时预览**。这些"工件"让你能快速判断结果对不对，而不需要全程盯着过程。这也是并行运行多个 Agent 的前提——你不可能同时盯三个 Agent 的实时输出，但你可以在它们都完成后，花几分钟浏览三份工件。

**5. 并行而非串行**

一次只跑一个 Agent，然后坐在那里等它完成，本质上还是人在卡节奏。更好的方式是同时启动多个 Agent，让它们并行处理不同的任务或同一个任务的不同方案。Cursor 的 `/best-of-n` 功能甚至可以让多个模型同时跑同一个任务，然后你从结果里挑最好的。

这改变了"效率"的定义：不是单次迭代更快，而是**单位时间内并行的迭代更多**。

## 人真正的价值在哪里

想清楚范式之后，我对"人的价值"这件事也有了新的理解。

人不应该是过程中的reviewer，而应该是：

- **定义问题的人**：什么是值得解决的问题，优先级怎么排
- **设计评价标准的人**：成功长什么样，用什么衡量
- **识别跑偏的人**：AI的迭代方向对不对，有没有在优化错误的指标

这三件事目前AI还替代不了，因为它们需要对业务和用户的深度理解。

把精力集中在这里，人就不是瓶颈，而是杠杆。

Cursor 团队有句话说得好：开发者不再是用来写代码的，而是帮助搭建"生产软件的工厂"的。你的价值不在于亲手砌每一块砖，而在于设计这座工厂的蓝图，并确保流水线不会跑偏。

## 还有多少时间

这件事的紧迫性可能比大多数人想的要高。

Cursor 的数据：2025 年 3 月，Tab 用户数量还是 Agent 用户的 2.5 倍。几个月后，情况就反转了——Agent 用户变成了 Tab 用户的 2 倍。过去一年里 Agent 使用量增长超过 15 倍。

Tab 时代持续了将近两年。同步 Agent 时代，他们预判撑不到一年。

这意味着什么？意味着**留给我们适应新范式的窗口期很短**。现在不开始思考"怎么让 Agent 自主工作"，等第三个时代全面到来的时候，你连焦虑的资格都没有了——因为你已经不在这个生产循环里了。

## 结尾

如果非要总结一句话的感悟：

**以前我们焦虑的是"AI会不会抢走我的工作"，现在我焦虑的是"我会不会拖累AI的工作"。**

这个转变，比我预想的来得更快。而从 Cursor 的实践来看，这不是某个人的感觉，而是整个行业正在发生的结构性变化。唯一的问题是：你是在搭建工厂，还是还在手工砌砖？
